# E.V.I.E. Scanning and Integration Implementation Todo Plan

Date: 2026-03-28
Owner: GreenReach Platform Team
Scope: Improve E.V.I.E. device scanning and onboarding across wired and wireless protocols, with first-class Light Engine bus mapping workflow integration.

> Update notice (2026-04-24): Scanning remains the onboarding flow for **devices and bus channels**. For **crop operations** (create planting, override targets, record harvest, repeat planting) E.V.I.E. now uses **group-first verbs** and must not require tray scans. See `docs/features/GROUP_LEVEL_MANAGEMENT_UPDATES.md` section 4.7.

Key architecture constraints applied:

- Farm is cloud-only. LE-EB is the farm runtime.
- Central is the user-facing hub. E.V.I.E. tool calls route through Central to LE.
- No unrestricted terminal or superuser behavior for E.V.I.E.
- Human-in-the-loop confirmation is required for registration and write operations.

## 2) Current Capability Snapshot

E.V.I.E. already has:

- Tooling in Central for get_device_status, scan_devices, register_device.
- LE discovery endpoint at /discovery/scan aggregating SwitchBot, Kasa, network, BLE, and MQTT stages.
- Central proxy for discovery endpoints and bus endpoints:
  - GET /api/bus-mappings
  - POST /api/bus-mapping
  - GET /api/bus/:busId/scan
- Bus mapping wizard in dashboard UI with scan, identify, assign, model validation, and save flow.

## 3) Problem Statement

E.V.I.E. does not yet provide a single guided onboarding flow that:

1. Scans wireless devices and wired channels together.
2. Distinguishes discovered versus registered inventory in user language.
3. Moves directly from detection to assignment using Light Engine bus mapping.
4. Produces auditable, low-friction onboarding with explicit user confirmation.

## 4) Target User Workflow (End State)

1. User asks E.V.I.E. to discover new devices.
2. E.V.I.E. runs a unified scan profile:
   - Wireless: SwitchBot cloud, Kasa, BLE, mDNS or LAN discovery where applicable.
   - Wired: Bus scan by bus id and node or channel inventory.
3. E.V.I.E. returns grouped findings:
   - New wireless devices.
   - New wired channels not mapped to groups.
   - Already-registered assets.
4. E.V.I.E. guides setup path per item:
   - Wireless device registration and room or zone assignment.
   - Wired channel identification and bus mapping assignment.
5. E.V.I.E. requests explicit confirmation for each write operation.
6. E.V.I.E. verifies final state by re-reading inventory and mappings.

## 5) Phased Implementation Todo

## Phase A: Contract and Data Model Alignment

A1. Define unified discovery result schema in Central
- Add a normalized schema for scan output with:
  - asset_kind: wireless_device or wired_channel
  - source: switchbot-cloud, kasa, ble, mqtt, mdns, bus-scan
  - discovered_id, display_name, protocol, bus_id, node_id, channel_id
  - registration_state: new, registered, mapped, unmapped
  - recommended_next_action
- Acceptance: scan payload from E.V.I.E. can be rendered without source-specific branching in prompts.

A2. Add mapping-aware inventory read tool
- Extend get_device_status response with bus mapping summary:
  - total_mapped_channels
  - unmapped_channels_by_bus
  - recent_mapping_updates
- Acceptance: one tool call can summarize both IoT inventory and bus mapping state.

A3. Introduce discovery session id
- Add a short-lived discovery_session_id generated by scan and reused by register and map actions.
- Acceptance: follow-up tool calls refer to discovered assets without retyping raw identifiers.

## Phase B: Tooling Extensions for E.V.I.E.

B1. Add scan_bus_channels tool
- Inputs: bus_id, optional deep_scan flag.
- Backend: call GET /api/bus/:busId/scan via existing Central proxy.
- Acceptance: E.V.I.E. can explicitly scan wired channels without UI dependency.

B2. Add get_bus_mappings tool
- Backend: call GET /api/bus-mappings.
- Acceptance: E.V.I.E. can explain existing mapping coverage and detect unmapped channels.

B3. Add save_bus_mapping tool (confirm tier)
- Backend: call POST /api/bus-mapping with required fields.
- Confirmation required before write.
- Acceptance: E.V.I.E. can complete wired mapping from chat with explicit approval.

B4. Upgrade scan_devices to unified scan mode
- Add mode parameter: wireless, wired, all.
- In all mode, aggregate wireless scan and bus scan findings in one response object.
- Acceptance: one E.V.I.E. command returns integrated wired and wireless findings.

## Phase C: Guided E.V.I.E. Conversation Flow

C1. Add deterministic orchestration flow in assistant instructions
- For discovery intent:
  - Step 1: run get_device_status.
  - Step 2: run unified scan mode all.
  - Step 3: summarize new assets by type and source.
  - Step 4: propose ordered onboarding actions.
- Acceptance: repeatable behavior across similar prompts.

C2. Add wired onboarding branch
- If unmapped bus channels exist, E.V.I.E. proposes mapping flow:
  - pick channel
  - optional identify action
  - assign group
  - model and spectrum metadata
  - confirm save
- Acceptance: E.V.I.E. can complete at least one channel mapping end-to-end.

C3. Add wireless onboarding branch
- For each new wireless device:
  - validate protocol data and naming
  - select room and zone
  - confirm register_device write
- Acceptance: E.V.I.E. can onboard multiple wireless devices in one session.

## Phase D: Light Engine Bus Mapping Tool Integration

D1. Add mapping suggestions in scan output
- Suggest group names using existing wizard naming logic.
- Include compatibility hints based on channel type.
- Acceptance: E.V.I.E. gives practical mapping suggestions before writes.

D2. Optional identify channel support
- Expose a safe identify tool wrapper for channel flash operation if endpoint is available.
- Acceptance: user can verify physical channel before saving mapping.

D3. Post-write sync event handling
- Ensure save_bus_mapping completion emits integration-ready state for dependent UI panels.
- Acceptance: mapping updates become visible without manual refresh loops.

## Phase E: Security, Permissions, and Audit

E1. Keep no-terminal-access posture
- Do not add shell, SSH, or unrestricted command tools.
- Acceptance: all onboarding actions remain scoped API operations.

E2. Confirmation policy
- Require explicit user confirmation for:
  - register_device
  - save_bus_mapping
  - any reassignment affecting existing mappings
- Acceptance: no write operation executes without clear user approval.

E3. Audit enhancements
- Log discovery sessions and write decisions with actor and farm context.
- Acceptance: each onboarding run is traceable from scan to final mapping.

## Phase F: Validation and Release Readiness

F1. Integration tests
- Test matrix:
  - wireless-only scan
  - wired-only scan
  - unified scan
  - mixed onboarding with register and bus mapping
- Acceptance: stable pass on all scenarios.

F2. Failure-path tests
- No credentials, empty scan, proxy timeout, partial bus scan, duplicate mapping id.
- Acceptance: E.V.I.E. returns actionable fallback guidance instead of generic errors.

F3. Documentation updates
- Update E.V.I.E. and IoT docs with final workflow examples and tool contracts.
- Acceptance: architecture, tools, and onboarding examples are internally consistent.

## 8) Success Criteria

- E.V.I.E. can discover and onboard both wired and wireless assets in one guided flow.
- At least 90 percent of new device onboarding can be completed without leaving E.V.I.E. chat.
- Bus mapping completion from E.V.I.E. produces same persisted result as dashboard wizard.
- All write operations are confirmed, auditable, and reversible where supported.
